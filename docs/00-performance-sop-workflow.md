### 1.2 优化原则

1. **不要过早优化**
   - 先保证代码正确性
   - 通过profiling确认瓶颈后再优化

2. **基于数据优化**
   - 使用监控工具收集数据
   - 优化前后进行对比

3. **渐进式优化**
   - 一次只优化一个点
   - 便于回滚和问题定位

4. **测试验证**
   - 生产环境优化前必须充分测试
   - 准备回滚方案

## 2. 性能问题发现

### 2.1 监控指标

| 指标类型 | 监控项 | 告警阈值 | 检查频率 |
|----------|--------|----------|----------|
| 响应时间 | P99延迟 | > 500ms | 实时 |
| 吞吐量 | QPS/TPS | 低于基线20% | 实时 |
| CPU | 使用率 | > 70% | 实时 |
| 内存 | 堆内存使用 | > 80% | 实时 |
| GC | Full GC频率 | > 1次/小时 | 实时 |
| 线程 | 线程数 | > 500 | 实时 |
| 数据库 | 连接池使用 | > 80% | 实时 |

### 2.2 问题发现渠道

**1. 监控系统告警**
```yaml
# 示例：Prometheus告警规则
- alert: HighResponseTime
  expr: http_request_duration_seconds{quantile="0.99"} > 0.5
  for: 5m
  labels:
    severity: warning
  annotations:
    summary: "响应时间过长"
```

**2. 日志分析**
```bash
# 慢查询日志
grep "Slow query" app.log | tail -100

# GC日志分析
java -jar gceasy.jar gc.log

# 慢接口统计
awk '{print $7}' access.log | sort | uniq -c | sort -rn | head -20
```

**3. 压测发现**
```bash
# JMeter压测
jmeter -n -t test_plan.jmx -l result.jtl

# 关注指标：
# - 吞吐量是否达标
# - 错误率是否可接受
# - 响应时间是否稳定
```

## 3. 性能问题定位

### 3.1 诊断工具

| 工具 | 用途 | 命令示例 |
|------|------|----------|
| Arthas | 实时诊断 | `java -jar arthas-boot.jar` |
| jstack | 线程分析 | `jstack -l <pid>` |
| jmap | 内存分析 | `jmap -dump:format=b,file=heap.hprof <pid>` |
| jstat | GC监控 | `jstat -gc <pid> 1000` |
| async-profiler | CPU分析 | `./profiler.sh -d 30 -f flame.html <pid>` |

### 3.2 诊断流程

**Step 1: 确认问题类型**
```bash
# 1. 查看进程资源占用
top -p <pid>

# 2. 线程CPU占用
ps -mp <pid> -o THREAD,tid,time | sort -rn

# 3. 内存使用
jmap -heap <pid>

# 4. GC情况
jstat -gcutil <pid> 1000
```

**Step 2: 定位热点代码**
```bash
# 使用async-profiler生成火焰图
./profiler.sh -d 30 -f flamegraph.html <pid>

# 热点方法分析
./profiler.sh -d 30 -t -f hotspots.txt <pid>
```

**Step 3: 分析线程状态**
```bash
# 导出线程栈
jstack -l <pid> > thread_dump.txt

# 分析线程状态
# - RUNNABLE: 正常执行
# - BLOCKED: 等待锁，可能存在锁竞争
# - WAITING/TIMED_WAITING: 等待条件，可能配置不合理
```

### 3.3 常见问题定位

| 现象 | 可能原因 | 排查方法 |
|------|----------|----------|
| CPU高 | 计算密集/死循环 | 火焰图分析 |
| 内存高 | 内存泄漏/大对象 | 堆转储分析 |
| 响应慢 | IO瓶颈/锁竞争 | 线程栈分析 |
| Full GC频繁 | 内存不足/对象生命周期长 | GC日志分析 |
| 连接池耗尽 | 连接泄漏/配置不当 | 连接池监控 |

## 4. 优化方案实施

### 4.1 优化决策矩阵

| 问题类型 | 优化策略 | 预期收益 | 风险等级 |
|----------|----------|----------|----------|
| 代码层面 | 算法优化/减少对象创建 | 中等 | 低 |
| JVM配置 | GC算法/内存参数调整 | 中等 | 中 |
| SQL优化 | 索引/查询优化 | 高 | 低 |
| 缓存引入 | 本地/分布式缓存 | 高 | 中 |
| 并发优化 | 线程池/锁优化 | 中等 | 中 |
| 架构调整 | 异步/分库分表 | 高 | 高 |

### 4.2 优化检查清单

**代码层面优化**
- [ ] 是否选择了合适的集合类？
- [ ] 字符串拼接是否使用了StringBuilder？
- [ ] 循环内部是否避免了对象创建？
- [ ] 是否避免了自动装箱/拆箱？
- [ ] 是否存在重复计算？

**JVM优化**
- [ ] 堆内存设置是否合理（Xms=Xmx）？
- [ ] GC算法是否适合应用场景？
- [ ] 是否配置了OOM处理？
- [ ] 是否启用了逃逸分析？

**数据库优化**
- [ ] 连接池配置是否合理？
- [ ] SQL是否有索引支持？
- [ ] 是否使用了批量操作？
- [ ] 是否避免了N+1查询？

**并发优化**
- [ ] 线程池参数是否配置正确？
- [ ] 锁的粒度是否足够小？
- [ ] 是否使用了合适的并发容器？
- [ ] 是否存在死锁风险？

**缓存优化**
- [ ] 是否处理了缓存穿透？
- [ ] 是否处理了缓存击穿？
- [ ] 是否处理了缓存雪崩？
- [ ] 缓存一致性策略是否明确？

### 4.3 实施注意事项

1. **小步快跑**
   - 每次只修改一处
   - 独立测试验证
   - 便于回滚

2. **保留证据**
   - 优化前的性能数据
   - 代码版本备份
   - 配置变更记录

3. **灰度验证**
   - 先在测试环境验证
   - 再在生产环境灰度
   - 观察后再全量

## 5. 优化效果验证

### 5.1 验证项目

| 验证项 | 通过标准 | 验证方法 |
|--------|----------|----------|
| 功能正确性 | 无回归bug | 回归测试 |
| 性能提升 | 达到预期目标 | 压测对比 |
| 稳定性 | 无异常/崩溃 | 稳定性测试 |
| 资源使用 | CPU/内存正常 | 监控观察 |

### 5.2 性能对比报告模板

```markdown
# 性能优化报告

## 优化背景
- 问题描述：接口响应时间过长，P99达到2s
- 优化目标：P99降至500ms以内

## 优化方案
1. 引入Redis缓存热点数据
2. SQL添加索引优化
3. 调整线程池参数

## 优化结果
| 指标 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| P99延迟 | 2000ms | 300ms | 85% |
| QPS | 1000 | 5000 | 400% |
| CPU使用率 | 80% | 40% | 50% |

## 风险评估
- 引入缓存可能导致数据不一致，已采用Cache-Aside模式
- 已配置缓存过期时间和刷新机制

## 上线计划
1. 灰度发布（10%流量）- 2024-01-15
2. 观察24小时后全量
```

### 5.3 回滚条件

出现以下情况立即回滚：
- 错误率上升超过0.1%
- 响应时间反而增加
- 出现OOM或其他严重异常
- 业务功能异常

## 6. 生产环境上线

### 6.1 上线流程

```
1. 准备阶段
   ├── 确认优化代码已合并
   ├── 准备回滚脚本
   └── 通知相关人员

2. 灰度发布
   ├── 发布10%节点
   ├── 观察15分钟
   ├── 对比核心指标
   └── 确认无异常后扩大灰度

3. 全量发布
   ├── 分批发布（每次30%）
   ├── 每批观察10分钟
   └── 全部发布完成

4. 监控观察
   ├── 观察24小时核心指标
   ├── 收集用户反馈
   └── 确认优化效果
```

### 6.2 监控重点

**发布后1小时内**
- 错误率变化
- 响应时间变化
- 系统资源使用

**发布后24小时内**
- 业务指标波动
- 用户反馈
- 长尾延迟

### 6.3 应急响应

| 场景 | 响应措施 | 负责人 |
|------|----------|--------|
| 错误率突增 | 立即回滚 | 值班人员 |
| 响应时间增加 | 评估后回滚 | 开发人员 |
| OOM/崩溃 | 自动回滚 + 告警 | 值班人员 |
| 数据不一致 | 暂停服务 + 数据修复 | 开发人员 |
